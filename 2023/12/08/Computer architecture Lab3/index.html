ILP

Based on the results, when we adjust parameters like decode width, issue width, and commit width, the total cycle performance fluctuates. If the performance varies a lot with these changes, it suggests the benchmark has high ILP. This means the processor can effectively use multiple instructions at once. If performance remains stable despite changes in these parameters, it indicates low ILP, suggesting the processor struggles to use instructions effectively.

Compare with go.ss

the pc.ss benchmark shows higher cycle counts and lower sensitivity to parameter changes suggests a lower degree of ILP compared to the go.ss benchmark.

For pc.ss variations in parameters, such as decode width, issue width, etc., have a smaller impact on the overall performance. So, ILP is lower.

Best performance/cost trade-off

consider a smaller ILP and aim for the lowest values for the sum of all parameters

set all the configuration parameters to their minimum values.

  • -decode:width 1
  • -issue:width 1
  • -commit:width 1
  • -res:ialu 1
  • -res:imult 1
  • -res:fpalu 1
  • -res:fpmult 1
